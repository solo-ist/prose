# ADR: Feature Flags with Opt-In Access for Gated Integrations

**Status:** Partially Adopted
**Date:** 2026-03-19
**Deciders:** Angel (maintainer)

> **v1.0 Decision:** Settings-persisted flags only. HMAC-signed access codes, CLI tooling, and remote config are deferred ‚Äî the threat model (users unlocking features on their own app) doesn't warrant crypto infrastructure at this scale. Flags live in `~/.prose/settings.json` under `featureFlags` and are gated via reactive Zustand selectors in `src/renderer/lib/featureFlags.ts`. See the implementation for details.

## Context

Prose has two integrations ‚Äî Google Docs and reMarkable ‚Äî that are fully implemented but hidden behind static placeholder UI ("Coming in v1.1"). The current gating is a hard-coded conditional in each component with no dynamic control.

As Prose moves toward releasing these features, we need a system that lets us:

1. **Gate features by default** ‚Äî new installs don't see unreleased integrations.
2. **Allow users to request access** ‚Äî a visible opt-in flow so interested users can raise their hand.
3. **Let the maintainer manually enable features** for specific users without shipping a new build.
4. **Eventually do a general rollout** ‚Äî flip a switch to make a feature available to everyone.

### Constraints

- **No user identification system exists.** No accounts, device IDs, licensing, or telemetry. Privacy is a core value.
- **Electron desktop app.** Settings are local (`~/.prose/settings.json`), and the app may run offline.
- **Small user base in alpha.** The system should be proportionate ‚Äî no enterprise-grade flag infrastructure.
- **No backend server.** Prose is a standalone desktop app with no first-party API.

## Decision

**Use local feature flags in settings, unlocked via HMAC-signed access codes, with an optional static remote config for broad rollouts.**

The system has three layers:

1. **Local feature flag registry** ‚Äî a `featureFlags` object in `Settings` that gates UI and IPC access.
2. **Signed unlock codes** ‚Äî HMAC-SHA256 tokens encoding a feature name, generated by the maintainer and entered by users in a settings field. Validated entirely client-side using a public verification key embedded in the app.
3. **Remote config (optional, phase 2)** ‚Äî a static JSON file hosted on GitHub Pages or a CDN that the app fetches on launch to check for broad rollout flags. No user data is sent ‚Äî it's a simple GET of a public file.

### How it works end-to-end

**User requests access:**
1. User sees a gated feature in Settings with a "Request Access" link ‚Üí opens a GitHub issue (pre-filled template) or a simple web form.
2. User also sees an "I have an access code" input field.

**Maintainer grants access:**
1. Angel runs a local CLI tool (a Node script in `scripts/`) that generates an HMAC-signed code for a specific feature (e.g., `google-docs`).
2. Angel sends the code to the user via email, GitHub issue reply, or DM.
3. The code is a base64url-encoded string like `Z29vZ2xlLWRvY3M6MTc0MjQzMjAwMDoxYTJiM2M0ZDVl`.

**User activates:**
1. User pastes the code into the "Access code" field in Settings.
2. The app decodes and verifies the HMAC signature using a public key embedded in the binary.
3. On success, `featureFlags.googleDocs` is set to `true` and persisted. The code itself is also stored so the flag survives settings resets.
4. The integration UI becomes fully interactive.

**General rollout:**
1. Angel updates the remote config JSON to set `googleDocs: true`.
2. On next launch, the app fetches the config, sees the flag is globally enabled, and unlocks it for everyone ‚Äî no code needed.
3. The `featureFlags` field in settings is updated to reflect the new default.

## Options Considered

### Option A: Signed Unlock Codes (Recommended)

| Dimension | Assessment |
|-----------|------------|
| Complexity | Medium ‚Äî HMAC signing is ~50 LOC, UI changes are straightforward |
| Cost | Zero ‚Äî no infrastructure, no accounts, no server |
| Privacy | Excellent ‚Äî no user tracking, no network calls for basic flow |
| Scalability | Good for dozens of users; awkward beyond hundreds |
| Team familiarity | High ‚Äî standard Node.js crypto, no new dependencies |
| Shareability risk | Low ‚Äî codes don't contain secrets; sharing a code just means another user gets access, which is fine at alpha scale |

**Pros:**
- Zero infrastructure cost. Works offline. No user accounts needed.
- The maintainer has full control over who gets access and when.
- Codes can optionally include expiry timestamps for time-limited beta access.
- Privacy-preserving: no telemetry, no phone-home, no device fingerprinting.
- Can layer remote config on top later without changing the local model.

**Cons:**
- Codes are manually generated and distributed ‚Äî doesn't scale to thousands of users.
- Codes can be shared (acceptable for alpha; mitigated by making codes non-secret).
- Requires embedding a verification key in the app binary (can be extracted, but the threat model is low ‚Äî users "hacking" their own app to unlock features is fine).

### Option B: Remote Feature Flag Service (LaunchDarkly, Flagsmith, PostHog)

| Dimension | Assessment |
|-----------|------------|
| Complexity | High ‚Äî requires user identification, SDK integration, account setup |
| Cost | Free tier available, but adds ongoing dependency |
| Privacy | Poor ‚Äî requires sending device/user identifiers to a third party |
| Scalability | Excellent ‚Äî built for millions of users |
| Team familiarity | Low ‚Äî new tooling, new dashboard, new failure mode |

**Pros:**
- Industry-standard approach. Rich targeting (percentage rollouts, cohorts, A/B tests).
- Real-time flag updates without app restart.
- Audit trail of who has what flags.

**Cons:**
- Requires inventing a user/device identifier, which Prose deliberately doesn't have.
- Adds a network dependency ‚Äî features break if the service is down and the app hasn't cached flags.
- Massive overkill for an alpha with a handful of users.
- Conflicts with Prose's privacy-first stance.

### Option C: GitHub-Based Allowlist

| Dimension | Assessment |
|-----------|------------|
| Complexity | Low-Medium ‚Äî fetch a JSON allowlist from a GitHub Gist or repo |
| Cost | Zero |
| Privacy | Moderate ‚Äî requires some stable identifier to match against the list |
| Scalability | Fine for small scale; editing a JSON file for each user is tedious |
| Team familiarity | High |

**Pros:**
- Simple to implement. GitHub is already central to the workflow.
- Easy to audit who's on the list.
- No new infrastructure.

**Cons:**
- Still requires a user identifier (email? GitHub username?). Where does the app get this?
- Network-dependent ‚Äî offline users can't activate features.
- Requires the user to enter their identifier in the app, which feels like account creation.
- Race condition between Gist updates and app fetches.

### Option D: Compile-Time Build Variants

| Dimension | Assessment |
|-----------|------------|
| Complexity | Low ‚Äî just flip a constant per build |
| Cost | Zero |
| Privacy | Perfect ‚Äî no runtime checks, no network |
| Scalability | Terrible ‚Äî can't grant access without shipping a new build |
| Team familiarity | High ‚Äî already used for `__IS_MAS_BUILD__` |

**Pros:**
- Simplest possible implementation. Already a pattern in the codebase.
- No runtime overhead.

**Cons:**
- Can't enable features for individual users without separate builds.
- The "request access" flow becomes "wait for the next release."
- Doesn't solve the actual problem.

## Trade-off Analysis

The core tension is between **privacy/simplicity** and **scalability/automation**.

Option B (remote service) is the right choice for a product with thousands of users and a user account system. Prose has neither, and adding both just for feature flags inverts the priority ‚Äî you'd be building infrastructure to support the flag system rather than the flag system supporting the product.

Option C (GitHub allowlist) is appealing for its simplicity but breaks down at the "how does the app identify itself" step. Without user accounts, there's no natural key to match against.

Option D (build variants) doesn't solve the problem at all.

Option A (signed codes) hits the sweet spot: zero infrastructure, works offline, privacy-preserving, and gives the maintainer precise control. The "codes can be shared" concern is a non-issue at alpha scale ‚Äî and frankly, if a user wants to share access to a free alpha feature, that's word-of-mouth marketing.

The optional remote config layer (phase 2) covers the "flip a switch for everyone" use case without building a full flag service. A static JSON file on GitHub Pages is effectively free and infinitely cacheable.

## Implementation Sketch

### Settings Type Addition

```typescript
// src/renderer/types/index.ts
interface Settings {
  // ... existing fields ...
  featureFlags?: {
    googleDocs?: boolean
    remarkable?: boolean
  }
  accessCodes?: string[]  // Stored codes for persistence across settings resets
}
```

### Feature Flag Module

```
src/shared/featureFlags.ts     ‚Äî Flag definitions, defaults, code verification
src/main/featureFlags.ts       ‚Äî IPC handlers for flag checks and code redemption
scripts/generate-access-code.ts ‚Äî CLI tool for maintainer to generate codes
```

### Code Format

```
base64url( featureName + ":" + expiryTimestamp + ":" + hmacSignature )
```

- `featureName`: `google-docs`, `remarkable`, `*` (wildcard for all)
- `expiryTimestamp`: Unix seconds, or `0` for no expiry
- `hmacSignature`: First 10 bytes of HMAC-SHA256(featureName + ":" + expiry, secret)

### UI Changes

Each gated feature's placeholder component gains:
1. A "Request Access" button ‚Üí opens pre-filled GitHub issue or form
2. An "Enter Access Code" collapsible section with a text input and "Activate" button
3. On successful activation, the placeholder is replaced with the full integration UI

### Remote Config (Phase 2)

```json
// https://prose.dev/feature-flags.json (or GitHub Pages)
{
  "version": 1,
  "flags": {
    "googleDocs": false,
    "remarkable": false
  }
}
```

Fetched on app launch with a 24-hour local cache. If the remote flag is `true`, the feature is unlocked for everyone regardless of local settings. If fetch fails, fall back to local flags only.

## Consequences

**What becomes easier:**
- Shipping partially-complete features behind a gate without maintaining separate branches.
- Controlled beta testing with specific users before general availability.
- Gradual rollout of Google Docs and reMarkable integrations.
- Adding future gated features (same pattern, new flag name).

**What becomes harder:**
- Nothing significant. The system is additive and doesn't change existing architecture.

**What we'll need to revisit:**
- If Prose ever adds user accounts, the flag system should migrate to account-based targeting.
- If the user base grows to thousands, manual code generation becomes impractical ‚Äî at that point, consider a self-serve activation flow or a lightweight backend.
- The HMAC verification key embedded in the app is extractable. This is acceptable for the current threat model (users unlocking features on their own app), but shouldn't be used for anything security-critical.
